📘 GoF 디자인 패턴의 정의

디자인 패턴(Design Pattern)이란, 소프트웨어 설계에서 반복적으로 발생하는 문제에 대한 재사용 가능한 해법을 정형화한 것이다. 1994년 GoF(Gang of Four)가 23개의 디자인 패턴을 생성(5)·구조(7)·행위(11)로 체계화하였다.

💡 개념 해설

GoF 디자인 패턴은 객체지향 설계에서 검증된 "베스트 프랙티스"의 모음입니다. 개발자들 간의 공용 언어(Common Vocabulary) 역할을 하여, "이 부분은 옵저버 패턴으로 구현했어요"라고 하면 코드를 보지 않아도 대략적 구조를 파악할 수 있습니다.

반면 패턴의 오용이나 설계 원칙 위반은 안티 패턴(Anti-Pattern)이라는 기술 부채를 초래합니다. 올바른 적용과 경계를 함께 이해하는 것이 핵심입니다.

📘 안티 패턴의 정의

안티 패턴(Anti-Pattern)이란, 겉보기에는 합리적으로 보이지만 실제로는 소프트웨어 품질을 저하시키는 반복적인 나쁜 설계 관행이다. 1998년 Brown et al.이 체계화하였다.

⚖️ 디자인 패턴 vs 안티 패턴

구분디자인 패턴안티 패턴
성격좋은 해법 (Best Practice)나쁜 관행 (Worst Practice)
결과유지보수성·확장성 향상기술 부채·복잡도 증가
적용의도적 도입무의식적/무분별한 발생
해결패턴 적용으로 문제 해결리팩토링으로 제거

1.1 디자인 패턴의 개요

디자인 패턴(Design Pattern)이란, 소프트웨어 설계에서 반복적으로 발생하는 문제에 대한 재사용 가능한 해법을 정형화한 것이다.

1994년 Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides(이른바 Gang of Four, GoF)가 저서 "Design Patterns: Elements of Reusable Object-Oriented Software"에서 23개의 디자인 패턴을 체계화하였다.

디자인 패턴의 4대 구성요소:

구성요소설명
패턴 이름(Name)패턴을 식별하는 이름 (의사소통 도구)
문제(Problem)패턴이 해결하는 설계 문제
해법(Solution)클래스/객체의 구조와 상호작용
결과(Consequences)패턴 적용의 장단점과 트레이드오프

제2장 핵심 패턴의 적용 목적 및 UML 구조 (소문제 2)

2.1 싱글톤 패턴(Singleton) — 생성 패턴 대표

적용 목적: 시스템 전체에서 하나의 인스턴스만 존재해야 하는 경우 사용한다. 전역 접근점(Global Access Point)을 제공하면서 인스턴스 수를 제한한다.

제3장 안티 패턴의 정의 및 위험 사례 (소문제 3)

3.1 안티 패턴의 정의

안티 패턴(Anti-Pattern)이란, 겉보기에는 합리적으로 보이지만 실제로는 소프트웨어 품질을 저하시키는 반복적인 나쁜 설계 관행이다. 1998년 Brown et al.이 체계화하였다.

구분디자인 패턴안티 패턴
성격좋은 해법 (Best Practice)나쁜 관행 (Worst Practice)
결과유지보수성·확장성 향상기술 부채·복잡도 증가
적용의도적 도입무의식적/무분별한 발생
해결패턴 적용으로 문제 해결리팩토링으로 제거

3.2 대표적 안티 패턴 사례

[사례 1] God Object (신 객체)

항목내용
정의하나의 클래스가 너무 많은 책임(기능)을 집중적으로 담당하는 패턴
위반 원칙SRP(단일 책임 원칙)
증상수천 줄의 코드, 수십 개의 메서드, 모든 모듈이 이 클래스에 의존
위험성변경 시 광범위한 사이드 이펙트, 테스트 불가, 병렬 개발 불가
해결 방안클래스 분리(Extract Class), 책임 분배, Facade 패턴 적용

[사례 2] Spaghetti Code (스파게티 코드)

항목내용
정의구조화되지 않은 복잡한 코드로 제어 흐름이 얽혀 있는 상태
위반 원칙구조적 프로그래밍 원칙
증상goto문 남발, 깊은 중첩, 함수 간 상호 호출 고리, 전역 변수 의존
위험성코드 이해 불가, 디버깅 불가, 유지보수 비용 기하급수적 증가
해결 방안리팩토링(Extract Method), Strategy/Template Method 패턴

[사례 3] Golden Hammer (황금 망치)

항목내용
정의익숙한 기술/도구를 모든 문제에 무분별하게 적용하는 패턴
증상모든 데이터를 RDBMS에 저장, 모든 통신에 REST API 사용
해결폴리글랏(Polyglot) 접근 — 문제에 맞는 기술 선택

[사례 4] Lava Flow (용암 흐름)

항목내용

증상누가 만들었는지 모르는 코드, "건드리면 안 돼" 주석, 미사용 클래스
해결Dead Code 탐지 도구 활용, 코드 리뷰, 정기 리팩토링

3.3 안티 패턴 발생 원인과 예방 전략

발생 원인예방 전략
설계 없이 코딩 시작설계 리뷰(Design Review) 의무화
SOLID 원칙 미준수코드 리뷰 시 SOLID 체크리스트 적용

3.4 연관 이론 연결

SOLID 원칙과의 관계: 디자인 패턴은 SOLID 원칙을 구현하는 구체적 방법이다. SRP를 위반하면 God Object, OCP를 위반하면 경직된 설계, DIP를 위반하면 강한 결합이 발생한다.

리팩토링과의 관계: Martin Fowler의 "Refactoring"(1999)에서 제시한 코드 악취(Code Smell)는 안티 패턴의 전조 증상이다.

아키텍처 패턴과의 관계: GoF 패턴이 클래스/객체 수준이라면, MVC, MVP, MVVM은 아키텍처 수준의 패턴이다. Observer는 MVC의 핵심 패턴이며, Strategy는 DI의 기반이다.

안녕하세요, 여러분. 오늘은 소프트웨어공학의 클래식 중의 클래식, GoF 디자인 패턴과 안티 패턴을 다뤄볼 건데요. 이 토픽은 기술사 시험에서 정말 자주 나오는 단골 토픽이에요. 왜냐하면 객체지향 설계의 핵심을 묻는 토픽이고, UML 다이어그램을 그릴 수 있느냐를 검증하기에 딱 좋거든요. 출제위원 입장에서 23개 패턴이라는 명확한 범위가 있으니까 문제 만들기도 좋고요.

그리고 실무적으로도 디자인 패턴은 정말 중요해요. 여러분이 현업에서 프로젝트를 하다 보면, 선배 개발자가 "이건 팩토리 패턴으로 풀어야지" 이런 얘기를 하잖아요? 디자인 패턴은 개발자들 사이의 공용 언어 역할을 하거든요. "이 부분은 옵저버 패턴으로 구현했어요"라고 하면, 코드를 안 봐도 대략적인 구조가 머릿속에 그려지는 거예요. 그래서 패턴을 안다는 건 단순히 시험 공부를 넘어서, 실무 커뮤니케이션 능력까지 갖추는 거라고 볼 수 있어요.

GoF라는 건 "Gang of Four", 4인방이라는 뜻인데요. 1994년에 Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides 네 명이 "Design Patterns"라는 책을 출간했어요. 이 책이 소프트웨어 공학 역사에서 가장 영향력 있는 책 중 하나로 꼽히는데, 여기서 23개의 디자인 패턴을 체계화한 거예요.

소문제 구조부터 잡아볼게요. 이 문제는 전형적인 2교시형 소문제 3개 구조인데요. 소문제 1은 "23개 패턴을 어떻게 분류하느냐"라는 체계 이해예요. 소문제 2는 "각 분류의 대표 패턴을 UML로 그려라"라는 구체적 설계 능력이고요. 소문제 3은 "패턴을 잘못 쓰면 어떻게 되느냐"라는 경계의 메시지예요. "체계 → 구체화 → 경계"의 삼단 구조, 이 흐름을 서론에서 예고해 주면 출제위원이 "이 수험생은 문제의 의도를 파악했구나"라고 느끼게 됩니다.

소문제 1, GoF 3대 분류부터 갈게요. "생구행"으로 외우세요. 생성(Creational) 5개, 구조(Structural) 7개, 행위(Behavioral) 11개. 합치면 23개예요. 각 분류의 핵심 질문이 다른데요, 생성은 "객체를 어떻게 만들까?", 구조는 "객체를 어떻게 조합할까?", 행위는 "객체 간 어떻게 협력할까?"예요.

23종 전체를 표로 정리해야 하는데요, 시험장에서는 시간이 빠듯하잖아요? 이상적으로는 23종 전체를 표로 정리하는 게 가장 좋지만, 시간이 부족하면 각 분류별 대표 패턴 2~3개만이라도 목적을 정확히 쓰세요. 중요한 건 그냥 이름만 나열하는 게 아니라, 각 패턴이 "어떤 문제를 해결하느냐"를 한 줄씩 쓰는 거예요. 생성 패턴 5종은 "싱팩추빌프"로 외우세요. 싱글톤, 팩토리메서드, 추상팩토리, 빌더, 프로토타입.

소문제 2가 가장 중요한 부분이에요. 30% 배분하세요. 왜 30%냐면요, 여기서 UML 클래스 다이어그램을 직접 그려야 하기 때문이에요. 다이어그램을 그리면 답안지에서 시각적으로 돋보이거든요. 반드시 A4 용지에 손으로 직접 그려보는 연습을 해두세요. 최소 3번은 그려봐야 자연스럽게 나옵니다.

싱글톤부터 갈게요. 핵심이 세 가지예요. private 생성자, static 인스턴스 변수, public static getInstance() 메서드. 멀티스레드 환경에서의 구현 방식을 한 줄 넣으면 차별화 포인트가 돼요. Bill Pugh 방식이 추천되는데, static inner class를 활용하면 lazy loading과 thread safety를 동시에 보장할 수 있어요.

어댑터는 세 개의 박스를 그려야 해요. Client가 사용하는 Target 인터페이스, 기존의 Adaptee 클래스, 그리고 둘을 연결하는 Adapter 클래스. 실무에서 레거시 시스템을 통합할 때 이 패턴을 정말 많이 써요. REST API를 SOAP으로 변환하는 어댑터, XML을 JSON으로 변환하는 어댑터 같은 게 전부 이 패턴이에요.

옵저버는 네 개의 박스예요. Subject 인터페이스, ConcreteSubject, Observer 인터페이스, ConcreteObserver. 핵심은 Subject가 Observer 리스트를 가지고 있다가, 상태가 변경되면 리스트의 모든 Observer에게 notify하는 거예요. 실무에서 MVC 패턴, React의 상태 관리, Kafka의 Pub-Sub 메시징이 전부 옵저버 패턴의 변형이에요.

소문제 3, 안티 패턴이에요. 이 부분이 문제의 차별화 포인트예요. 안티 패턴은 "겉보기에 합리적이지만 실제로는 나쁜 설계 관행"이에요. "갓스골라"로 대표 4가지를 외우세요.

God Object가 가장 대표적이에요. 하나의 클래스가 모든 걸 다 하는 거예요. SRP(단일 책임 원칙)를 정면으로 위반하는 거죠. Spaghetti Code는 Y2K 때 COBOL 레거시 시스템이 가장 대표적인 사례예요. Golden Hammer는 "내가 가진 게 망치뿐이면 모든 게 못으로 보인다"는 거고, Lava Flow는 죽은 코드가 레거시처럼 남아있는 현상이에요.

중요한 건, 디자인 패턴의 과도한 도입도 안티 패턴이 될 수 있다는 거예요. YAGNI(You Ain't Gonna Need It) 원칙으로 방지해야 해요. 답안에 이 균형 관점을 쓰면 차별화됩니다.

결론에서는 "GoF 디자인 패턴은 SOLID 원칙을 구현하는 구체적 방법이며, 올바른 적용은 유지보수성을 향상시키지만 무분별한 도입은 안티 패턴이라는 기술 부채를 초래한다"고 정리하세요.

분량 배분은 서론 10%, 소문제 1이 25%, 소문제 2가 30%, 소문제 3이 25%, 결론 10%입니다. 필수 도식은 세 가지: 23종 분류표, 3개 패턴 UML 클래스 다이어그램, 안티 패턴 사례표.

시험장 팁: UML 다이어그램을 그릴 때 클래스 박스는 가로 3~4cm, 세로 2~3cm 정도로 통일하면 깔끔해 보여요. 화살표는 반드시 의미를 표기하세요. 실선은 연관, 점선은 의존, 빈 삼각형은 상속, 채워진 다이아몬드는 합성이에요.

연관 토픽도 정리해 드릴게요. SOLID 원칙, 리팩토링, UML 다이어그램, MVC/MVP/MVVM 아키텍처 패턴과 직접 연결됩니다.

여러분, 디자인 패턴은 한 번에 다 외우려고 하지 마시고요, "생구행" 분류 → 대표 3개 패턴 UML → 안티패턴 4종, 이 순서로 반복하면 자연스럽게 체화됩니다. 오늘도 수고 많으셨습니다!

한 줄 정의

GoF 디자인 패턴은 객체지향 설계의 재사용 가능한 해법 23종을 생성(5)·구조(7)·행위(11)로 분류하며, 올바른 적용은 품질을 높이지만 무분별한 도입은 안티 패턴을 초래한다.

핵심 키워드 맵

[GoF 3대 분류]
├─ 생성(5): 싱글톤, 팩토리메서드, 추상팩토리, 빌더, 프로토타입
├─ 구조(7): 어댑터, 브릿지, 컴포지트, 데코레이터, 퍼사드, 플라이웨이트, 프록시
└─ 행위(11): 옵저버, 전략, 커맨드, 템플릿메서드, 이터레이터, 상태, 책임연쇄, 중재자, 메멘토, 비지터, 인터프리터
[핵심 패턴 UML]
├─ Singleton: private 생성자 + static getInstance()
├─ Adapter: Target ← Adapter → Adaptee
└─ Observer: Subject → notify → Observer.update()
[안티 패턴]
├─ God Object: SRP 위반, 클래스 비대
├─ Spaghetti Code: 구조화 부재
├─ Golden Hammer: 만능 기술 오류
└─ Lava Flow: 죽은 코드 방치
