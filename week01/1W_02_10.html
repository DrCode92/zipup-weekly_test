2026

📘 핵심 정의

옵티마이저(Optimizer)는 SQL 실행 계획을 결정하는 DBMS의 핵심 구성요소로, RBO(규칙 기반)에서 CBO(비용 기반)로 발전하였으며, 3대 조인 기법(NL Join, Sort Merge Join, Hash Join)의 선택은 데이터량·조인 조건·인덱스 유무에 따라 결정된다. Hash Join은 Build/Probe 2단계로 동작하며, 메모리 초과 시 Grace Hash Join으로 Temp Spill을 처리한다.

📚 개념 해설

본 토픽은 DBMS 쿼리 최적화의 핵심 메커니즘을 이해하고, 옵티마이저의 발전 과정, 조인 전략의 체계적 비교, Hash Join의 내부 동작을 심화 분석하는 능력을 검증합니다.

소문제 간 흐름: 최적화 전략(RBO→CBO) → 조인 기법 비교(NL/SM/Hash) → Hash Join 심화(Build/Probe, Temp Spill)

📖 원문 학습

제1장 옵티마이저의 발전 과정과 CBO 비용 산정

1.1 옵티마이저의 개요

옵티마이저(Optimizer)란, SQL 질의가 입력되었을 때 가장 효율적인 실행 계획(Execution Plan)을 결정하는 DBMS의 핵심 구성요소이다. SQL은 선언적(Declarative) 언어로, 사용자는 "무엇을(WHAT)" 원하는지만 명시하고 "어떻게(HOW)" 처리할지는 DBMS가 결정한다.

1.2 RBO(Rule-Based Optimizer)

RBO는 사전에 정의된 우선순위 규칙(Rule)에 따라 실행 계획을 결정하는 방식이다. Oracle RBO는 15단계 접근 경로 우선순위를 사용하였으나, 데이터 특성을 반영하지 못해 10g부터 비권장, 12c부터 완전 폐기되었다.

1.3 CBO(Cost-Based Optimizer)

CBO는 통계 정보를 기반으로 각 실행 계획의 예상 비용(Cost)을 산출하고, 최소 비용의 계획을 선택하는 방식이다. 비용 산정 5대 요소: 카디널리티, 선택도, I/O 비용, CPU 비용, 네트워크 비용.

제2장 3대 조인 기법 알고리즘 비교

2.1 Nested Loop Join

외부 테이블의 각 행에 대해 내부 테이블을 반복 탐색. O(N×M) / O(N×logM). 소량 데이터, OLTP에 적합.

2.2 Sort Merge Join

양쪽 정렬 후 병합. O(NlogN+MlogM+N+M). 범위 조건 가능. 대량 데이터에 유리.

2.3 Hash Join

Build Input으로 해시 테이블 생성, Probe Input 순차 스캔 매칭. O(N+M). 등호(=)만 가능.

제3장 Hash Join 심화 — Build/Probe Input과 Temp Spill

3.1 2단계 프로세스

[1단계] Build Phase: 작은 테이블로 인메모리 해시 테이블 구축. [2단계] Probe Phase: 큰 테이블 순차 스캔하며 해시 테이블 탐색.

3.2 Build/Probe 선정 기준

"작은 놈이 Build, 큰 놈이 Probe". 해시 테이블이 PGA Hash Area에 적재되어야 최적 성능.

3.3 Grace Hash Join

Build Input이 Hash Area 초과 시 양쪽을 N개 파티션으로 분할, Temp Tablespace에 저장 후 파티션 쌍별 메모리 내 Hash Join 수행. 2~10배 성능 저하.

안녕하세요, 여러분. 오늘은 데이터베이스의 꽃이라 할 수 있는 옵티마이저와 조인 기법을 다뤄볼 건데요. 이 토픽은 DB 출제 영역에서 단골 주제이고, 실무 경험을 답안에 녹여내기 좋은 토픽이에요.

소문제 구조: 소문제 1은 옵티마이저(RBO→CBO), 소문제 2는 3대 조인 비교, 소문제 3은 Hash Join 심화. "최적화 전략 → 조인 비교 → Hash Join 심화" 구조입니다.

소문제 1. RBO는 고정 규칙으로 실행 계획을 정하지만, 데이터 특성을 무시해요. CBO는 통계 기반으로 비용을 계산. 5대 요소를 "카선아씨네"로 외우세요. 중요한 건 통계 품질이에요.

소문제 2는 30% 비중으로 가장 커요. NL Join은 이중 루프, Sort Merge는 정렬+병합, Hash Join은 Build+Probe. 비교표 10개 기준이 핵심이에요. "네소해"로 기억하세요.

소문제 3. "작은 놈이 Build, 큰 놈이 Probe" — "빌프"로 외우세요. Temp Spill 발생 시 Grace Hash Join이 작동. Oracle에서는 SWAP_JOIN_INPUTS 힌트로 강제 지정 가능.

필수 도식 3종: 3대 조인 비교표, Hash Join 2단계도, Grace Hash Join 파티션 도식. "카선아씨네", "네소해", "빌프" — 이 세 두문자어면 뼈대가 잡힙니다!

한 줄 정의

CBO는 통계 기반 비용 산정으로 최적 실행 계획을 결정하며, 3대 조인(NL/SM/Hash) 선택은 데이터량·조인 조건·인덱스 유무에 따라 달라진다.

핵심 키워드 맵

[옵티마이저]
├─ RBO: 규칙 기반, 15단계, 데이터 무관
└─ CBO: 비용 기반, 통계 의존, 카선아씨네
[3대 조인]
├─ NL: 중첩 루프, O(N×logM), 소량+인덱스
├─ SM: 정렬+병합, O(NlogN), 범위 조건
└─ Hash: 빌드+프로브, O(N+M), 대량+등호
[Hash Join 심화]
├─ Build: 작은 테이블 → 해시 테이블
├─ Probe: 큰 테이블 → 순차 스캔
└─ Spill: 메모리 초과 → Grace Hash Join
