📘 핵심 정의

VM(Virtual Machine)은 하이퍼바이저를 통해 하드웨어를 추상화하여 독립된 Guest OS를 제공하는 하드웨어 수준 가상화 기술이며, 컨테이너(Container)는 리눅스 커널의 Namespaces/Cgroups를 활용한 경량 프로세스 격리 기술이고, 쿠버네티스(Kubernetes)는 대규모 컨테이너를 선언적으로 오케스트레이션하는 클라우드 네이티브 표준 플랫폼이다.

📚 개념 해설

본 토픽은 가상화 기술의 구조적 차이를 이해하고, 리눅스 커널 메커니즘 기반의 컨테이너 격리 원리를 설명하며, 쿠버네티스의 오케스트레이션 아키텍처를 체계적으로 도식화할 수 있는 능력을 검증합니다.

제1장 VM과 컨테이너의 구조적 비교

1.1 가상화 기술의 진화 배경

가상화 기술은 1960년대 IBM의 메인프레임 시분할 시스템에서 시작되어, 물리적 자원의 효율적 활용이라는 근본적 과제를 해결하기 위해 발전해 왔다. 1999년 VMware가 x86 아키텍처에서 최초의 상용 하이퍼바이저를 출시하면서 서버 가상화 시대가 본격화되었으며, 2013년 Docker의 등장은 가상화의 패러다임을 OS 수준에서 프로세스 수준으로 전환시키는 혁명적 전환점이 되었다.

1.2 VM 아키텍처

VM은 하이퍼바이저(Hypervisor)를 통해 하드웨어를 추상화하여 독립된 가상 머신을 제공하는 기술이다. Type 1(Bare-metal)은 하드웨어 위에 직접 설치되는 방식(VMware ESXi, KVM, Hyper-V), Type 2(Hosted)는 호스트 OS 위에 설치되는 방식(VMware Workstation, VirtualBox)이다.

1.3 컨테이너 아키텍처

컨테이너는 호스트 OS의 커널을 공유하면서 프로세스 수준의 격리를 제공하는 경량 가상화 기술이다. Guest OS가 없이 호스트 커널을 직접 공유하기 때문에, VM 대비 부팅 시간이 수초 이내로 빠르며, 메모리 사용량도 MB 단위로 경량이다.

Namespaces는 "무엇을 볼 수 있는가(가시성)"를 제어하고, Cgroups는 "얼마나 사용할 수 있는가(자원 한도)"를 제어한다.

2.4 통합 동작 흐름

1단계: clone() 시스템 콜 (CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS) → 2단계: 새 Namespace 세트 생성 → 3단계: Cgroup 프로세스 등록 (memory.limit, cpu.quota) → 4단계: seccomp + AppArmor 적용 → 5단계: 컨테이너 프로세스 실행

3.4 Pod 배포 흐름

3.5 Reconciliation Loop

선언적 모델: Desired State ↔ Current State를 Controller Manager가 지속 비교 → 차이 발생 시 자동 조정 → Self-healing. 이것이 kubectl apply(선언형)를 사용하는 이유.

소문제 간 통합 관점

"추상화 수준의 계층적 심화": VM vs 컨테이너 이해 → 커널 수준 격리 심화 → 대규모 오케스트레이션. 이는 "가상화의 진화(HW → OS → 컨테이너 오케스트레이션)" 기술 발전 흐름과 일치한다.

연관 이론: MSA(컨테이너/K8s의 동인), DevOps/CI/CD(빌드-테스트-배포 파이프라인의 핵심 단위), 서버리스(Knative 기반 K8s 위에 구현)

안녕하세요, 여러분. 오늘은 CA/OS 영역에서 정말 중요한 토픽, VM과 컨테이너, 그리고 쿠버네티스를 다뤄볼 거예요. 이 토픽은 기술사 시험에서 거의 매회 출제된다고 봐도 과언이 아닌데요, 클라우드 네이티브가 대세가 되면서 출제 빈도가 더 높아지고 있어요.

현업에서도 이 토픽은 아주 실용적이에요. 프로젝트 인프라를 설계할 때 "이 워크로드는 VM이 적합한가, 컨테이너가 적합한가?"를 판단해야 하잖아요? 금융권 레거시 시스템은 보안 격리 때문에 VM이 필수인 경우가 많고, MSA 기반 신규 시스템은 컨테이너가 효율적이에요.

소문제 구조를 잡을게요. 소문제 1은 "VM과 컨테이너의 구조적 차이를 비교하라", 소문제 2는 "그래서 컨테이너가 어떻게 격리를 하느냐", 소문제 3은 "그 컨테이너를 대규모로 운영하려면 어떻게 해야 하느냐". "비교 → 기반 기술 → 대규모 운영", 이 삼단 구조를 서론에 딱 한 줄 써주면 출제위원이 "이 사람은 문제 전체의 맥락을 이해하고 있구나"라고 느끼게 됩니다.

소문제 1부터 갈게요. VM과 컨테이너의 가장 결정적인 차이는 Guest OS의 유무예요. VM은 하이퍼바이저 위에 각각 독립된 Guest OS를 올리기 때문에 이미지 크기가 수 GB, 부팅에 수 분이 걸려요. 반면 컨테이너는 호스트 커널을 공유해서 이미지 수십~수백 MB, 부팅 수 초. 10개 기준 비교표를 반드시 만드세요.

하이퍼바이저 Type 1과 Type 2의 차이도 한 줄 넣어주세요. Type 1은 ESXi, KVM 같은 Bare-metal 방식, Type 2는 VirtualBox 같은 Hosted 방식이에요.

소문제 2가 핵심이에요. 30% 배분. 컨테이너의 격리는 Namespaces와 Cgroups 두 가지예요. Namespaces는 "무엇을 볼 수 있는가", Cgroups는 "얼마나 사용할 수 있는가".

Namespaces 6종은 "피넷마유아이"로 외우세요. PID, NET, MNT, UTS, IPC, USER. clone() 시스템 콜에 CLONE_NEWPID 같은 플래그를 전달하면 커널이 새 네임스페이스를 만들어요.

Cgroups 5대 컨트롤러는 "씨메블씨피"로 외우세요. CPU, Memory, blkio, cpuset, pids. memory.limit_in_bytes를 초과하면 OOM Killer가 발동하는 메커니즘까지 설명하면 깊이가 달라져요.

소문제 3, 쿠버네티스. 아키텍처 다이어그램이 가장 중요해요. Control Plane 4대 컴포넌트는 "에에스컨"으로 외우세요. API Server, etcd, Scheduler, Controller Manager.

etcd는 Raft 합의 알고리즘으로 동작하고 홀수 노드(3, 5, 7)로 구성하는 게 정석이에요. Worker Node에는 kubelet, kube-proxy, Container Runtime. K8s 1.24부터 Docker 직접 미지원, containerd/CRI-O 사용하는 점도 최신 트렌드예요.

핵심 원리는 Reconciliation Loop. 선언적 모델로 Desired State ↔ Current State를 계속 비교하며 자동 조정. 이게 Self-healing이에요.

결론: "가상화가 HW 가상화 → OS 가상화 → 컨테이너 오케스트레이션으로 진화해 왔으며, MSA, DevOps, 서버리스로 이어지는 클라우드 네이티브 전환의 핵심 인프라"라고 정리하세요.

필수 도식 3종: VM/컨테이너 아키텍처 비교도, Namespaces 6종+Cgroups 구조도, K8s 클러스터 아키텍처도. 분량 배분: 서론 10%, 소문제1 25%, 소문제2 30%, 소문제3 25%, 결론 10%.

여러분, "피넷마유아이", "씨메블씨피", "에에스컨" — 이 세 두문자어만 확실히 외우면 이 토픽은 절반 이상 끝난 거예요. 오늘도 수고 많으셨습니다!

한 줄 정의

VM은 하드웨어 수준 가상화를 통해 완전한 OS 격리를 제공하고, 컨테이너는 리눅스 커널의 Namespaces/Cgroups를 활용한 경량 프로세스 격리 기술이며, 쿠버네티스는 대규모 컨테이너를 선언적으로 오케스트레이션하는 클라우드 네이티브 표준 플랫폼이다.

핵심 키워드 맵

[VM vs 컨테이너]
├─ 하이퍼바이저 (Type 1/2) · Guest OS 유무 · 격리 수준 · 부팅 시간 · 집적도
[Namespaces/Cgroups]
├─ NS 6종: PID, NET, MNT, UTS, IPC, USER
├─ CG 컨트롤러: CPU, Memory, blkio, cpuset, pids
└─ clone()/unshare() 시스템 콜
[K8s 아키텍처]
├─ Control Plane: apiserver, etcd(Raft), scheduler, controller-mgr
├─ Worker Node: kubelet, kube-proxy, Container Runtime
└─ Reconciliation Loop (선언적 모델)
