인터럽트(Interrupt) 처리 메커니즘과 PCB 2026

개요 및 개념

인터럽트(Interrupt)란 CPU가 현재 실행 중인 명령어 처리를 일시 중단하고, 긴급하거나 우선적인 처리가 필요한 사건(Event)에 대응하기 위해 제어권을 특정 서비스 루틴(ISR)으로 이전하는 비동기적 이벤트 처리 메커니즘이다.

PCB(Process Control Block)는 운영체제가 각 프로세스를 관리하기 위해 유지하는 프로세스별 메타데이터 자료구조로, 문맥 교환(Context Switching) 시 프로세스의 실행 상태를 완전히 보존하고 복원하는 핵심 단위이다.

왜 인터럽트가 필요한가?

인터럽트가 없는 시스템에서는 CPU가 I/O 장치의 작업 완료를 반복적으로 확인하는 폴링(Polling) 방식을 사용해야 합니다. 이는 CPU 자원을 극도로 낭비합니다. 인터럽트 방식은 I/O 장치가 작업 완료 시 CPU에 신호를 보내는 방식으로, CPU는 그 사이 다른 유용한 작업을 수행할 수 있습니다.

❌ 폴링(Polling) 방식

CPU가 "끝났어? 끝났어?" 반복 확인. CPU 시간의 최대 90%를 I/O 대기에 소모. CPU 자원 극도로 낭비.

✅ 인터럽트(Interrupt) 방식

I/O 장치가 "끝났어!" 신호 전송. CPU 대기 시간 2~5%로 감소. 현대 OS의 멀티태스킹 기반.

최적화 기법 (클릭하여 펼치기)

Top Half / Bottom Half 분리— ISR을 긴급/지연 처리로 분리
▼

ISR을 긴급 처리(Top Half)와 지연 처리(Bottom Half)로 분리한다. Top Half는 최소한의 긴급 작업만 수행하고, 나머지는 Bottom Half로 미룬다. Linux에서 softirq, tasklet, workqueue가 이 Bottom Half 메커니즘이다. 인터럽트 비활성화 시간을 최소화하여 시스템 응답성을 높인다.

Threaded Interrupt— ISR을 커널 스레드로 실행
▼

ISR을 커널 스레드로 실행하여 선점 가능하게 만든다. 우선순위가 높은 다른 인터럽트에 의해 중단될 수 있어, 실시간 시스템에서 유연한 우선순위 제어가 가능하다.

Interrupt Coalescing— 다수 인터럽트를 묶어 처리
▼

NIC(네트워크 카드)에서 패킷마다 인터럽트를 걸면 초당 수만 번의 인터럽트가 발생한다. Coalescing은 일정 수의 패킷 또는 일정 시간 단위로 묶어서 한 번에 처리하여 인터럽트 빈도를 획기적으로 줄인다.

NAPI (Linux)— 인터럽트 + 폴링 혼합 방식
▼

New API. 평상시에는 인터럽트 방식을 사용하다가, 고부하 시 폴링 방식으로 전환하여 인터럽트 스톰(Interrupt Storm)을 방지한다. Linux 네트워크 서브시스템의 핵심 최적화 기법이다.

연관 이론 연결

DMA와의 관계

DMA 컨트롤러는 CPU 개입 없이 I/O↔메모리 직접 전송. 전송 완료 시 CPU에 인터럽트 발생. 두 개념은 상호 보완적.

시스템 콜과의 관계

시스템 콜은 소프트웨어 인터럽트(Trap)로 구현. INT 0x80 또는 SYSCALL → 동일한 모드 전환, 컨텍스트 저장 메커니즘 작동.

프로세스 상태 전이

타이머 인터럽트 → Running→Ready (선점). I/O 완료 인터럽트 → Blocked→Ready. 인터럽트는 프로세스 스케줄링의 물리적 기반.

제1장 개요 및 배경

1.1 인터럽트의 정의와 필요성

인터럽트(Interrupt)란 CPU가 현재 실행 중인 명령어 처리를 일시 중단하고, 긴급하거나 우선적인 처리가 필요한 사건(Event)에 대응하기 위해 제어권을 특정 서비스 루틴으로 이전하는 메커니즘이다. 인터럽트는 운영체제 커널의 핵심 동작 원리이자, 현대 컴퓨터 시스템의 비동기적 이벤트 처리(Asynchronous Event Handling)를 가능하게 하는 근본 메커니즘이다.

인터럽트가 없는 시스템에서는 CPU가 I/O 장치의 작업 완료를 지속적으로 확인하는 폴링(Polling) 방식을 사용해야 한다. 이 방식은 CPU가 I/O 완료 여부를 반복적으로 검사하므로 CPU 자원을 극도로 낭비한다. Intel의 벤치마크에 따르면, 폴링 방식은 CPU 시간의 최대 90%를 I/O 대기에 소모할 수 있으며, 인터럽트 방식은 이를 2~5% 수준으로 감소시킨다.

1.2 인터럽트의 분류

인터럽트는 발생 원인과 특성에 따라 다음과 같이 분류된다.

분류 기준유형설명예시
발생 원인하드웨어 인터럽트 (External)CPU 외부의 하드웨어 장치가 발생시키는 신호키보드 입력, 디스크 I/O 완료, 타이머 만료, 네트워크 패킷 도착
소프트웨어 인터럽트 (Internal)CPU 내부에서 명령어 실행 중 발생하는 사건0으로 나누기, 오버플로우, 시스템 콜(INT 0x80)
처리 시점동기적 (Synchronous)명령어 실행의 직접적 결과로 발생트랩(Trap), 폴트(Fault), 시스템 콜
비동기적 (Asynchronous)현재 실행과 무관하게 외부에서 발생I/O 완료, 타이머, 하드웨어 오류
복귀 가능성복귀 가능 (Recoverable)원인 해결 후 중단 지점으로 복귀 가능페이지 폴트 → 페이지 로드 후 재실행
복귀 불가 (Non-recoverable)시스템 중단 또는 프로세스 종료하드웨어 오류, 보호 위반(Protection Fault)
답안 작성 핵심: 인터럽트를 단순히 "하드웨어/소프트웨어"로만 구분하면 합격권이다. 동기/비동기, 복귀 가능/불가능까지 함께 분류하면 깊이가 달라진다. 두문자어 "하소동비복"으로 3가지 분류 축을 암기한다.

제3장 인터럽트 처리 과정 (단계별 시간순)

인터럽트 발생부터 처리 완료까지의 전체 과정을 8단계로 정리한다. 이 과정은 하드웨어 자동 처리 단계와 소프트웨어(OS 커널) 처리 단계로 구분된다.

3.1 전체 처리 흐름도

프로세스 A 실행 중
↓
── H/W 자동 수행 ──
① 인터럽트 발생 (IRQ 신호)
↓
② CPU: 현재 명령어 완료 대기
↓
③ CPU: 인터럽트 인식 및 확인 (INTA)
↓
④ CPU: PSW, PC를 커널 스택에 자동 저장
↓
⑤ CPU: 모드 전환 (User→Kernel), 인터럽트 비활성화
↓
── S/W (OS 커널) 수행 ──
⑥ OS: 나머지 컨텍스트(GPR 등) PCB에 저장
↓
⑦ OS: ISR(Interrupt Service Routine) 실행
↓
⑧ OS: 스케줄러 호출 → 다음 프로세스 결정
↓

↓

3.2 각 단계 상세

[1단계] 인터럽트 발생

하드웨어 장치가 인터럽트 요청 라인(IRQ)에 전기 신호를 인가한다. PIC(Programmable Interrupt Controller)가 이 신호를 수신하여 우선순위를 판별한 후, CPU의 INT 핀에 인터럽트 신호를 전달한다. 현대 x86 시스템에서는 APIC(Advanced PIC)가 이 역할을 수행하며, MSI(Message Signaled Interrupt) 방식으로 메모리 쓰기를 통해 인터럽트를 전달하기도 한다.

[2단계] 현재 명령어 완료 대기

CPU는 현재 실행 중인 명령어를 완료한 후에야 인터럽트를 처리한다. 이는 CPU의 원자적(Atomic) 명령어 실행 보장을 위함이다. 단, NMI(Non-Maskable Interrupt)는 이 규칙의 예외로, 하드웨어 치명적 오류 시 즉시 처리된다.

[3단계] 인터럽트 인식 및 확인 (INTA)

CPU는 PSW의 인터럽트 허용 비트(IF, Interrupt Flag)를 검사한다. IF=1이면 인터럽트를 수용하고, PIC에 INTA(Interrupt Acknowledge) 신호를 보낸다. PIC는 인터럽트 벡터 번호를 데이터 버스에 실어 CPU에 전달한다.

[4단계] PSW, PC 자동 저장 (하드웨어 수행)

CPU는 하드웨어적으로(자동으로) 현재의 PSW와 PC 값을 커널 스택에 push한다. 이 단계는 소프트웨어의 개입 없이 CPU 마이크로코드에 의해 수행되며, 인터럽트 처리의 원자성을 보장하는 핵심 메커니즘이다.

저장 항목저장 위치목적
PC (Program Counter)커널 스택복귀 주소 보존 (중단된 명령어의 다음 주소)
PSW (Program Status Word)커널 스택CPU 상태(모드 비트, 조건 코드, IF) 보존

[5단계] 모드 전환 및 인터럽트 비활성화

CPU는 PSW의 모드 비트를 커널 모드(Supervisor Mode)로 전환하고, 인터럽트 허용 플래그(IF)를 0으로 설정하여 추가 인터럽트를 일시적으로 차단한다. 동시에 IDT(Interrupt Descriptor Table)에서 인터럽트 벡터 번호에 해당하는 ISR의 시작 주소를 읽어 PC에 로드한다.

[6단계] 나머지 컨텍스트 저장 (소프트웨어/OS 수행)

ISR의 초입부에서 운영체제 커널이 소프트웨어적으로 범용 레지스터(GPR), 스택 포인터(SP) 등 나머지 CPU 레지스터 값을 현재 프로세스의 PCB(Process Control Block)에 저장한다. 이 과정이 문맥 저장(Context Save)이다.

저장 항목저장 위치방법
범용 레지스터 (R0~R15)PCB의 레지스터 영역PUSH/MOV 명령어 (소프트웨어)
스택 포인터 (SP)PCB의 SP 필드MOV 명령어
부동소수점 레지스터PCB의 FPU 영역FXSAVE 명령어 (x86)
기타 제어 레지스터PCB의 제어 영역MOV 명령어

[7단계] ISR(Interrupt Service Routine) 실행

인터럽트의 원인에 해당하는 서비스 루틴이 실행된다. 예를 들어, 디스크 I/O 완료 인터럽트라면 DMA 전송 완료를 확인하고, 대기 중인 프로세스를 Ready Queue로 이동시킨다. 타이머 인터럽트라면 시간 퀀텀(Time Quantum) 만료를 확인하고 스케줄러를 호출한다.

[8단계] 스케줄러 호출 및 컨텍스트 복원 (문맥 교환)

ISR 처리 완료 후, 스케줄러가 호출되어 다음에 실행할 프로세스를 결정한다. 선택된 프로세스의 PCB에서 레지스터 값들을 CPU에 복원하고(Context Restore), IRET(Interrupt Return) 명령어로 PSW와 PC를 복원하면 해당 프로세스가 실행을 재개한다.

제4장 PCB(Process Control Block) 상세

4.1 PCB의 정의와 역할

PCB(Process Control Block)는 운영체제가 각 프로세스를 관리하기 위해 유지하는 프로세스별 메타데이터 자료구조이다. 프로세스의 생성 시 할당되고, 종료 시 해제된다. PCB는 커널 메모리 영역에 저장되며, 사용자 프로세스는 직접 접근할 수 없다.

PCB의 핵심 역할은 문맥 교환(Context Switching) 시 프로세스의 실행 상태를 완전히 보존하고 복원하는 것이다. PCB 없이는 멀티태스킹이 불가능하다.

4.3 PCB와 문맥 교환(Context Switching)

문맥 교환은 CPU가 현재 프로세스(P₁)의 실행을 중단하고 다른 프로세스(P₂)의 실행을 재개하는 과정이다. 이 과정에서 PCB가 핵심 역할을 수행한다.

P₁ 실행 중 P₂ 대기 중
↓ 인터럽트 발생
P₁의 CPU 상태 → PCB₁에 저장
↓
스케줄러: P₂ 선택
↓
PCB₂에서 CPU 상태 → CPU에 복원
↓
P₁ Ready 대기 P₂ 실행 재개

문맥 교환 비용: 문맥 교환은 순수한 오버헤드(Pure Overhead)이다. 문맥 교환이 진행되는 동안 CPU는 유용한 작업을 수행하지 않는다. 일반적으로 문맥 교환에 소요되는 시간은 1~10마이크로초(μs) 수준이며, 이는 프로세서 속도, PCB 크기, 캐시 무효화(Cache Invalidation) 비용에 따라 달라진다.

문맥 교환 비용 요소설명영향
레지스터 저장/복원GPR, FPU 등 수십 개 레지스터 I/O직접적 시간 비용 (~1μs)
TLB 플러시 (TLB Flush)프로세스 전환 시 TLB 무효화이후 메모리 접근 지연 (수십 μs 누적)
캐시 오염 (Cache Pollution)L1/L2 캐시의 유효 데이터가 새 프로세스 데이터로 대체Cold Start로 인한 캐시 미스 증가
파이프라인 플러시CPU 파이프라인의 진행 중인 명령어 폐기파이프라인 재충전 비용

제5장 성능 분석 및 최적화

5.1 인터럽트 지연 시간 (Interrupt Latency)

인터럽트 지연 시간은 인터럽트 발생부터 ISR 첫 번째 명령어 실행까지의 시간이다. 실시간 운영체제(RTOS)에서는 이 지연 시간의 보장이 핵심 요구사항이다.

구간소요 시간 (일반적)결정 요인
인터럽트 인식수 ns ~ 수십 nsPIC/APIC 처리 속도
명령어 완료 대기0 ~ 수백 ns현재 실행 중인 명령어 복잡도
PSW/PC 자동 저장수십 ns스택 메모리 접근 속도
ISR 진입수십 ~ 수백 nsIDT 참조, 분기 예측
전체 지연0.5 ~ 10 μs시스템에 따라 차이

RTOS vs 범용 OS 비교: Linux의 인터럽트 지연은 수십 μs 수준이나, VxWorks 같은 RTOS는 수 μs 이내를 보장한다. PREEMPT_RT 패치를 적용한 Linux는 50μs 이내의 최악 지연 시간(Worst-case Latency)을 달성할 수 있다.

제6장 연관 이론 연결

6.1 DMA(Direct Memory Access)와의 관계

DMA 컨트롤러는 CPU 개입 없이 I/O 장치와 메모리 간 데이터를 직접 전송한다. 전송 완료 시 DMA 컨트롤러가 CPU에 인터럽트를 발생시켜 완료를 알린다. 따라서 DMA는 인터럽트 처리 메커니즘을 전제로 동작하며, 두 개념은 상호 보완적이다.

6.2 시스템 콜(System Call)과의 관계

시스템 콜은 사용자 프로그램이 커널 서비스를 요청하는 메커니즘으로, 소프트웨어 인터럽트(Trap)를 통해 구현된다. Linux에서 INT 0x80 또는 SYSCALL 명령어가 트랩을 발생시키면, 인터럽트와 동일한 메커니즘으로 모드 전환, 컨텍스트 저장, 커널 코드 실행이 수행된다.

6.3 프로세스 상태 전이도와의 관계

인터럽트는 프로세스 상태 전이의 주요 트리거이다. 타이머 인터럽트는 Running → Ready 전이를 유발하고(선점), I/O 완료 인터럽트는 Blocked → Ready 전이를 유발한다. 따라서 인터럽트 처리 메커니즘은 프로세스 스케줄링의 물리적 기반이다.

제7장 시험 출제 관점

기출 연계 포인트: 인터럽트 처리와 PCB는 1교시 단독 출제는 물론, 문맥 교환, 프로세스 스케줄링, I/O 관리 등과 결합하여 2교시에서도 출제된다.

차별화 전략: ① 하드웨어 자동 저장(PSW, PC)과 소프트웨어 저장(GPR → PCB)의 구분을 명확히 하고, ② 문맥 교환 비용의 구체적 요소(TLB 플러시, 캐시 오염)를 서술하며, ③ Top Half/Bottom Half 분리 같은 최적화 기법까지 언급하면 합격권을 넘는 답안이 된다.

분량 비중 가이드 (1교시 답안 기준): 서론 15% / 본론 75% / 결론 10%

인터럽트가 왜 필요한가

자, 먼저 인터럽트가 왜 필요한지부터 봅시다. 인터럽트가 없으면 CPU가 I/O 장치에 "끝났어? 끝났어? 끝났어?" 이렇게 계속 물어봐야 해요. 이걸 폴링이라고 하는데요, CPU가 대기하면서 아무 일도 못 하거든요. Intel 벤치마크에 따르면 CPU 시간의 최대 90%를 이런 대기에 쓸 수 있어요. 말도 안 되는 낭비잖아요?

인터럽트 방식은 이걸 뒤집어요. I/O 장치가 일 끝나면 "나 끝났어!"라고 CPU한테 신호를 보내는 거예요. CPU는 그때까지 다른 일을 하다가, 신호가 오면 그때 처리하면 됩니다. 이러면 CPU 대기 시간이 2~5%로 줄어요. 엄청난 차이죠.

8단계 처리 과정 — 순서대로 따라가 봅시다

자, 인터럽트 처리 과정을 8단계로 쪼개서 순서대로 따라가 봅시다. 이걸 "발완인저전컨실복"으로 외우세요.

1단계, 발생. 하드웨어 장치가 IRQ 라인에 전기 신호를 보내요. 키보드를 누르면 키보드 컨트롤러가 IRQ1에 신호를 보내는 식이죠. PIC가 이 신호를 받아서 우선순위를 판단하고 CPU에 전달해요.

2단계, 완료 대기. CPU는 현재 실행 중인 명령어를 끝까지 마무리해요. 명령어 하나를 반쯤 실행하다가 그냥 중단할 수는 없거든요. 원자성이 보장되어야 하니까요.

3단계, 인식. CPU가 PSW의 인터럽트 플래그(IF)를 확인해요. IF=1이면 "좋아, 인터럽트 받겠어" 하고 PIC한테 INTA 신호를 보내요. PIC가 벡터 번호를 알려주죠.

4단계, 여기가 핵심이에요. 저장. CPU가 하드웨어적으로, 자동으로, PSW와 PC를 커널 스택에 push해요. 소프트웨어가 개입하는 게 아니라 CPU 마이크로코드가 하는 거예요. 이게 왜 중요하냐면, 이 두 값이 안 저장되면 원래 실행하던 프로그램으로 돌아갈 수 없거든요.

5단계, 전환. 사용자 모드에서 커널 모드로 전환되고, IF를 0으로 만들어서 추가 인터럽트를 일시적으로 차단해요. 그리고 IDT에서 ISR 주소를 읽어서 PC에 넣어요.

6단계, 컨텍스트 저장. 여기서부터는 소프트웨어, 즉 OS 커널이 하는 일이에요. 범용 레지스터, 스택 포인터, 부동소수점 레지스터 같은 나머지 CPU 상태를 PCB에 저장해요. 답안에서 꼭 강조해야 할 게, 4단계는 하드웨어가, 6단계는 소프트웨어가 한다는 거예요. 이 구분을 명확하게 쓰면 채점관 눈에 확 들어옵니다.

7단계, ISR 실행. 실제 인터럽트 원인을 처리하는 거죠. 디스크 I/O 완료면 데이터 전송 확인, 타이머 만료면 스케줄러 호출.

8단계, 복원. 스케줄러가 다음 프로세스를 정하고, 그 프로세스의 PCB에서 레지스터 값을 CPU에 복원해요. IRET 명령어로 PSW, PC를 복원하면 끝. 해당 프로세스가 마치 중단된 적이 없는 것처럼 실행을 재개하는 거예요.

🎬 [핵심 설명 — PCB와 문맥 교환 (약 7분)]

PCB가 뭔가

자, 이제 PCB 이야기를 합시다. PCB는 Process Control Block, 프로세스 제어 블록이에요.

쉽게 비유하면 프로세스의 "이력서"라고 생각하시면 돼요. 이 프로세스가 누군지(PID), 부모가 누군지(PPID), 지금 뭐 하고 있었는지(PC, 레지스터 값들), 현재 상태가 뭔지(Running/Ready/Blocked), 메모리는 어디를 쓰고 있는지, 파일은 뭘 열어놨는지. 이 모든 정보가 PCB에 다 들어있어요. 프로세스가 생성되면 PCB가 만들어지고, 프로세스가 끝나면 PCB도 해제됩니다.

PCB가 왜 중요하냐면, 이게 없으면 문맥 교환이 안 돼요. 문맥 교환이 안 되면 멀티태스킹이 안 되고, 멀티태스킹이 안 되면 현대 운영체제가 존재할 수 없어요. 그만큼 핵심적인 자료구조라는 거죠.

답안에 PCB 필드를 쓸 때는 반드시 표로 정리하세요. 크게 다섯 그룹으로 나누면 돼요. "식레상메스"로 외우세요. 식별정보(PID, PPID), 레지스터(PC, SP, PSW, GPR), 상태(State, Priority), 메모리(페이지 테이블 포인터), 스케줄링(CPU 사용시간, 대기시간). 이 다섯 그룹을 행으로 놓고 각각의 구체적 필드를 적어주면 깔끔한 표가 됩니다.

여기서 주의할 점이 있어요. 많이들 PCB 필드 중에서 레지스터 부분만 강조하는데, 사실 메모리 관리 필드도 중요해요. 특히 "페이지 테이블 포인터"가 있잖아요? x86에서는 CR3 레지스터가 이 역할을 하는데, 문맥 교환 시 CR3를 바꾸면 가상 메모리 공간 전체가 바뀌는 거예요. 이게 TLB 플러시를 유발하는 원인이기도 하고요. 이 연결 관계를 이해하면 문맥 교환의 비용 구조가 확 이해됩니다.

문맥 교환의 비용

문맥 교환 이야기를 좀 더 해볼게요. 문맥 교환은 순수한 오버헤드예요. 문맥 교환하는 동안에는 유용한 일을 아무것도 못 하거든요. 보통 1~10마이크로초 걸린다고 하는데, 진짜 비용은 그게 아니에요.

진짜 비용은 뭐냐면, TLB 플러시와 캐시 오염이에요. 프로세스가 바뀌면 TLB(Translation Lookaside Buffer)를 무효화해야 하잖아요? 가상 메모리 매핑이 다르니까요. 그러면 새 프로세스가 시작할 때 TLB 미스가 잔뜩 나요. 캐시도 마찬가지예요. L1, L2 캐시에 이전 프로세스 데이터가 가득 차 있는데, 새 프로세스는 그 데이터가 소용없잖아요? 결국 캐시 미스가 연쇄적으로 발생하는 거죠. 이걸 캐시 워밍업, 또는 콜드 스타트라고 해요.

이 부분을 답안에 쓰면 확실히 차별화가 됩니다. 대부분의 수험생이 "문맥 교환은 레지스터 저장/복원이다" 정도만 쓰거든요. TLB 플러시, 캐시 오염, 파이프라인 플러시까지 쓰면 깊이가 달라져요.

그러면 이 비용을 줄이려면 어떻게 해야 할까요? 현대 OS에서는 몇 가지 최적화를 써요. 첫째, Top Half와 Bottom Half 분리예요. 인터럽트 처리 루틴을 두 부분으로 나누는 거예요. Top Half는 정말 급한 것만 빨리 처리하고, 나머지는 Bottom Half로 미루는 거죠. Linux에서 softirq, tasklet, workqueue가 이 Bottom Half 메커니즘이에요. 이렇게 하면 인터럽트가 비활성화되어 있는 시간을 최소화할 수 있어요.

둘째, Interrupt Coalescing이라는 기법이 있어요. 네트워크 카드에서 패킷이 올 때마다 인터럽트를 걸면, 초당 수만 번의 인터럽트가 발생하잖아요? 이걸 모아서 한 번에 처리하는 거예요. 인터럽트 빈도를 획기적으로 줄일 수 있죠. 이런 최적화 기법을 답안 결론부에 한 줄이라도 쓰면 실무적 관점이 인정됩니다.

🎬 [답안 작성 팁 — 약 2분]

실전에서 이렇게 쓰세요

서론에서는 인터럽트 정의를 쓰고, 폴링과 인터럽트 방식을 간략히 비교하세요. "CPU 자원 활용의 효율성을 위해 인터럽트 방식이 사용된다" 정도로 연결하면 됩니다.

본론에서는 흐름도를 반드시 그리세요. 8단계를 수직으로 나열하고, 하드웨어 수행 단계와 소프트웨어 수행 단계를 점선으로 구분해주세요. 그 다음에 PCB 필드를 표로 정리하고, 문맥 교환 과정을 P₁→P₂ 전환 다이어그램으로 보여주면 완벽합니다.

결론에서는 DMA와 시스템 콜과의 관계를 간략히 언급하고, Top Half/Bottom Half 분리 같은 최적화 기법을 한 줄이라도 쓰면 시의성이 확보됩니다.

🎬 [마무리 — 약 2분]

자, 오늘 내용 정리합니다.

암기 포인트 다시 한번 갈게요. 인터럽트 처리 8단계는 "발완인저전컨실복". 발생, 완료대기, 인식, 저장, 전환, 컨텍스트, ISR실행, 복원. PCB 핵심 필드 그룹은 "식레상메스". 식별, 레지스터, 상태, 메모리, 스케줄링. 문맥 교환 비용 4요소는 "레TL캐파". 레지스터 저장/복원, TLB 플러시, 캐시 오염, 파이프라인 플러시.

그리고 이 토픽에서 가장 중요한 포인트 하나만 더 강조할게요. PSW/PC는 하드웨어가 자동으로 저장하고, GPR은 소프트웨어(OS 커널)가 저장한다. 이 구분을 명확하게 쓰면 "CPU의 마이크로 수준 동작을 정확히 이해하고 있다"는 인상을 줄 수 있어요. 이게 출제의도가 묻는 핵심이기도 하고요.

이 토픽은 프로세스 스케줄링, I/O 관리, DMA와 직접 연결됩니다. 특히 이번 주 모의고사에 나오는 스케줄링 문제랑 연계가 돼요. 타이머 인터럽트가 선점 스케줄링의 물리적 기반이거든요. 타이머 인터럽트 없이는 RR, MLFQ 같은 선점형 알고리즘이 동작할 수 없어요. 이런 식으로 토픽 간 연결고리를 의식하면서 공부하시면 훨씬 효과적입니다.

여러분, 인터럽트가 어려워 보여도 결국은 "저장하고 처리하고 복원한다"는 단순한 흐름이에요. 이 뼈대를 잡아놓고 디테일을 채워나가면 반드시 정복할 수 있습니다.

마지막으로 한 가지 더 말씀드리면, 이 토픽은 실무에서도 정말 중요해요. 임베디드 시스템 개발이나 OS 커널 튜닝에서 인터럽트 지연시간을 줄이는 게 핵심 과제거든요. 최근에는 RTOS(Real-Time OS)에서 인터럽트 응답 시간을 마이크로초 단위로 보장하는 게 자율주행이나 의료기기 같은 안전 필수 시스템의 핵심 요구사항이에요. 이런 현대적 맥락까지 알아두시면 시험에서도 실무에서도 큰 도움이 될 겁니다.

6. 차별화 포인트

합격권 답안을 넘는 서술 포인트

  • PSW/PC의 하드웨어 자동 저장과 GPR의 소프트웨어 저장을 명확히 구분하면 "CPU 동작을 마이크로 수준으로 이해하고 있다"는 인상을 준다.
  • 문맥 교환 비용에서 TLB 플러시와 캐시 오염을 구체적으로 언급하면 합격권을 넘는 깊이를 확보할 수 있다.
  • Top Half/Bottom Half 분리를 Linux 커널의 실제 구현(softirq, tasklet, workqueue)과 연결하면 실무적 관점이 강화된다.
  • DMA, 시스템 콜과의 관계를 제시하면 인터럽트가 OS 전체 아키텍처에서 차지하는 위치를 보여줄 수 있다.
