명세 기반 테스트, 구조 기반 테스트 커버리지, MC/DC 2026

명세 기반 테스트(Specification-based Testing)는 소프트웨어의 내부 구조를 알지 못하는 상태에서 요구사항 명세서, 설계 문서 등 외부 명세를 기반으로 테스트 케이스를 도출하는 블랙박스 테스트 기법이다.

구조 기반 테스트(Structure-based Testing)는 소프트웨어의 내부 코드 구조(소스 코드, 제어 흐름)를 분석하여 테스트 케이스를 설계하는 화이트박스 테스트 기법으로, 코드의 커버리지(Coverage)가 핵심 지표이다.

MC/DC(Modified Condition/Decision Coverage)는 복합 조건식에서 각 개별 조건이 전체 결정 결과에 독립적으로 영향을 미침을 증명하여 마스킹 효과를 방지하는 최상위 커버리지 기준이다.

개념 해설

소프트웨어 테스트 설계 기법은 크게 명세 기반(블랙박스)과 구조 기반(화이트박스)으로 나뉩니다. 명세 기반은 "무엇을 테스트하는가"에 초점을 맞춰 사용자 관점에서 입력과 출력의 관계를 검증하고, 구조 기반은 "얼마나 테스트했는가"를 코드 커버리지로 측정합니다.

특히 항공, 자동차, 의료기기 같은 Safety-Critical System에서는 단순 커버리지를 넘어 각 조건이 실제로 결정 결과에 영향을 미쳤는지를 수학적으로 증명하는 MC/DC가 필수적으로 요구됩니다.

모든 실행 가능한 문장이 최소 한 번 실행되었는지 측정 — 최저 수준

▼

산출 공식: 구문 커버리지(%) = (실행된 문장 수 / 전체 실행 가능 문장 수) × 100

1: if (A > 0)    ← 조건문
2:   X = X + 1  ← 문장 1
3: if (B > 0)    ← 조건문
4:   Y = Y + 1  ← 문장 2

TC {A=1, B=1}로 모든 문장 실행 → 구문 커버리지 100%. 그러나 A≤0이거나 B≤0인 경우의 경로는 검증 못 함.

2

분기 커버리지 (Branch / Decision Coverage)

모든 분기의 True/False 경로가 최소 한 번 실행 — 분기 ⊃ 구문

▼

산출 공식: 분기 커버리지(%) = (실행된 분기 수 / 전체 분기 수) × 100

위 예시에서 분기 커버리지 100% 달성:

TC1: {A=1, B=1} → 두 조건 모두 True
TC2: {A=-1, B=-1} → 두 조건 모두 False

포함 관계: 분기 커버리지 100% → 구문 커버리지 100% 자동 충족. 역은 성립하지 않음.

각 조건의 독립적 영향 증명 — 최상위, N+1개 TC

▼

마스킹 효과 (Masking Effect)

복합 조건식에서 하나의 조건 값이 다른 조건의 결정 결과에 대한 영향을 차폐(Masking)하는 현상입니다.

AND 연산 마스킹

D = A AND B

A = False이면 B가 뭐든 D = False

→ False가 마스킹

OR 연산 마스킹

D = A OR B

A = True이면 B가 뭐든 D = True

→ True가 마스킹

논리식 증명: D = A AND (B OR C) — A=False일 때 마스킹

거짓 안전감 (False Sense of Security)

A=False 상태에서 B, C를 어떻게 바꿔도 D=False → B, C의 결함을 전혀 검출 못 함. 단순 커버리지 100%가 실질적 검증을 보장하지 않습니다.

제1장 명세 기반 테스트 주요 기법 (소문제 1)

1.1 명세 기반 테스트 개요

명세 기반 테스트(Specification-based Testing)는 소프트웨어의 내부 구조를 알지 못하는 상태에서 요구사항 명세서, 설계 문서 등 외부 명세를 기반으로 테스트 케이스를 도출하는 기법이다. 블랙박스 테스트(Black-box Testing)라고도 불리며, ISTQB에서 체계적으로 분류하고 있다.

명세 기반 테스트의 핵심 가치는 "사용자 관점"에서의 기능 검증에 있다. 코드 내부를 모르더라도 입력(Input)과 기대 출력(Expected Output)의 관계를 통해 결함을 발견할 수 있다.

1.2 동등 분할(Equivalence Partitioning)

동등 분할은 입력 데이터의 영역(Domain)을 동일한 처리가 예상되는 그룹(Partition)으로 분할하고, 각 그룹에서 대표값을 선택하여 테스트하는 기법이다. ISO/IEC/IEEE 29119-4에서 표준 기법으로 정의되어 있다.

원리: 동일 파티션 내의 모든 값은 동일한 결과를 산출할 것으로 기대되므로, 대표값 하나만 테스트해도 해당 파티션 전체를 검증한 것으로 간주한다.

예시: 성인 나이 입력 필드 (유효 범위: 19~65세)

파티션범위대표값기대 결과
유효 파티션19 ~ 6530정상 처리
무효 파티션 1 (하한 미만)~ 1810에러 메시지
무효 파티션 2 (상한 초과)66 ~80에러 메시지

1.3 경계값 분석(Boundary Value Analysis)

경계값 분석은 동등 분할의 파티션 경계(Boundary)에서 결함이 집중적으로 발생한다는 경험적 사실에 기반하여, 경계값과 그 인접값을 테스트하는 기법이다.

원리: Boris Beizer의 연구에 따르면, 소프트웨어 결함의 약 65~70%가 경계값 근처에서 발생한다. 이는 프로그래머가 ≤와 <, ≥와 >를 혼동하는 Off-by-One 오류가 빈번하기 때문이다.

예시: 동일 조건 (유효 범위: 19~65세)에서의 경계값 테스트

2.1 구조 기반 테스트 개요

구조 기반 테스트(Structure-based Testing)는 소프트웨어의 내부 코드 구조(소스 코드, 제어 흐름)를 분석하여 테스트 케이스를 설계하는 기법이다. 화이트박스 테스트(White-box Testing)라고도 하며, 코드의 얼마나 많은 부분이 실행되었는지를 측정하는 커버리지(Coverage)가 핵심 지표이다.

2.2 구문 커버리지(Statement Coverage)

구문 커버리지는 프로그램 내 모든 실행 가능한 문장(Statement)이 최소 한 번 이상 실행되었는지를 측정하는 가장 기본적인 커버리지 기준이다.

산출 공식: 구문 커버리지(%) = (실행된 문장 수 / 전체 실행 가능 문장 수) × 100

예시 코드:
1: if (A > 0)           ← 조건문
2:   X = X + 1          ← 문장 1
3: if (B > 0)           ← 조건문
4:   Y = Y + 1          ← 문장 2

테스트 케이스 {A=1, B=1}로 실행하면 모든 문장(1~4)이 실행되어 구문 커버리지 100%를 달성한다. 그러나 A≤0이거나 B≤0인 경우의 경로는 검증하지 못하는 한계가 있다.

2.3 분기 커버리지(Branch Coverage / Decision Coverage)

분기 커버리지는 프로그램 내 모든 분기(Branch)의 참(True)과 거짓(False) 경로가 최소 한 번 이상 실행되었는지를 측정한다.

산출 공식: 분기 커버리지(%) = (실행된 분기 수 / 전체 분기 수) × 100

위 예시에서 분기 커버리지 100%를 달성하려면:

  • 테스트 1: {A=1, B=1} → 두 조건 모두 True
  • 테스트 2: {A=-1, B=-1} → 두 조건 모두 False

분기 커버리지 ⊃ 구문 커버리지: 분기 커버리지 100%를 달성하면 구문 커버리지 100%도 자동으로 충족된다. 그 역은 성립하지 않는다.

2.4 조건 커버리지(Condition Coverage)

조건 커버리지는 복합 조건식 내의 각 개별 조건(Condition)이 True와 False를 모두 갖도록 하는 기준이다.

예시: if (A > 0 AND B > 0) 에서 개별 조건은 A > 0과 B > 0의 두 가지이다.

테스트A > 0B > 0조건 커버리지분기 결과
TC1: A=1, B=-1TFA:T, B:F ✓False
TC2: A=-1, B=1FTA:F, B:T ✓False

위 두 테스트로 조건 커버리지 100%를 달성하지만, 분기 결과가 모두 False여서 분기 커버리지는 50%에 불과하다. 이것이 조건 커버리지의 치명적 한계이다.

제3장 MC/DC 개념 및 마스킹 효과 방지 원리 (소문제 3)

3.1 MC/DC(Modified Condition/Decision Coverage) 개념

MC/DC란, 복합 조건식(Decision)에서 각 개별 조건(Condition)이 전체 결정(Decision) 결과에 독립적으로 영향을 미침을 증명하는 커버리지 기준이다. 1992년 Boeing사의 John Chilenski와 Steven Miller가 제안하였으며, FAA의 DO-178B/C 표준에서 Level A 소프트웨어의 필수 검증 기준으로 채택되었다.

MC/DC의 정의 (3가지 요건):

  • Decision Coverage: 전체 결정(Decision)이 True와 False를 모두 가져야 한다
  • Condition Coverage: 각 개별 조건이 True와 False를 모두 가져야 한다
  • 독립적 영향(Independent Effect): 각 조건이 결정 결과를 독립적으로 변경할 수 있어야 한다 — 다른 조건을 고정한 채, 해당 조건만 변경했을 때 결정 결과가 달라지는 테스트 쌍이 존재해야 한다

테스트 케이스 수: N개의 조건이 있을 때, MC/DC를 충족하는 최소 테스트 케이스 수는 N+1개이다. 이는 전체 조합(2N)보다 훨씬 적어, 실용적 수준에서 높은 커버리지를 보장한다.

결정식: D = A AND B (조건 2개)

  • B의 독립적 영향: TC1 vs TC2 (B: T→F, A 고정=T, D: T→F) ✓
  • 최소 3개(= 2+1) 테스트 케이스로 MC/DC 충족

3.3 마스킹 효과(Masking Effect)와 MC/DC의 방지 원리

마스킹 효과의 정의: 복합 조건식에서 하나의 조건 값이 다른 조건의 결정 결과에 대한 영향을 차폐(Masking)하는 현상이다.

마스킹 발생 메커니즘:

  • D = A AND B 에서: A = False이면, B가 True든 False든 D = False이다 → A = False가 B의 영향을 마스킹
  • D = A OR B 에서: A = True이면, B가 True든 False든 D = True이다 → A = True가 B의 영향을 마스킹

논리식을 통한 증명: D = A AND (B OR C) 에서

TCABCB OR CD
TC1FTTTF
TC2FTFTF
TC3FFTTF
TC4FFFFF

A = False인 상태에서는 B와 C를 어떻게 바꿔도 D = False이다. A가 B, C의 영향을 완전히 마스킹하고 있다. 단순 조건/분기 커버리지만으로는 이 마스킹 상태에서의 테스트 케이스를 채택하여 "커버리지 100%"를 달성할 수 있지만, 실제로는 B와 C의 결함을 전혀 검출하지 못하는 거짓 안전감(False Sense of Security)을 초래한다.

MC/DC의 마스킹 방지 원리: MC/DC는 각 조건의 독립적 영향 쌍에서 다른 조건이 마스킹되지 않는 상태를 요구한다.

MC/DC는 마스킹이 발생하지 않는 조건 조합에서만 독립적 영향 쌍을 인정하므로, 커버리지 100%라 하더라도 각 조건이 실제로 결정 결과에 영향을 미쳤음이 보장된다.

3.5 연관 이론 연결

V 모델(V-Model): 명세 기반 테스트는 V 모델의 좌측(요구분석↔인수테스트, 설계↔시스템테스트)에서 주로 활용되고, 구조 기반 테스트와 MC/DC는 V 모델의 하단(코딩↔단위테스트)에서 핵심적으로 적용된다.

결함 비용 곡선(Cost of Defect): Barry Boehm의 연구에 따르면 결함의 수정 비용은 단계가 진행될수록 기하급수적으로 증가하며, MC/DC 수준의 정밀한 단위 테스트는 초기 결함 제거를 통해 전체 프로젝트 비용을 크게 절감하는 효과가 있다.

소문제 1: 명세 기반 테스트

소문제 1부터 갈게요. 명세 기반 테스트, 다른 말로 블랙박스 테스트예요. 소프트웨어 내부 코드를 모르는 상태에서 요구사항 명세만 보고 테스트하는 거죠. 3가지 기법을 물어보는데, 동등 분할, 경계값 분석, 결정 테이블이에요. "동경결"로 외우세요.

동등 분할은 입력을 그룹으로 나누는 거예요. 학교에서 시험 성적을 A/B/C/D/F로 나누는 것과 비슷해요. 나이 입력 필드에서 19~65세가 유효 범위라면, 유효 파티션(19~65)과 무효 파티션 두 개(18 이하, 66 이상)로 나누는 거예요. 답안에는 반드시 파티션 구분 표를 그려야 해요.

경계값 분석은 동등 분할의 확장인데요, Boris Beizer라는 소프트웨어 테스팅 대가의 연구에 따르면 결함의 65~70%가 경계에서 발생한다고 해요. 이 숫자를 꼭 외워두세요. 왜 경계에서 결함이 많이 발생할까요? 프로그래머가 <=와 <를 헷갈려서 Off-by-One 오류를 내는 경우가 많기 때문이에요. ISTQB에서는 2-value BVA와 3-value BVA를 구분합니다.

결정 테이블은 조건이 여러 개 있을 때 조합을 표로 정리하는 거예요. 조건 2개면 최대 4개 규칙(22), 조건 3개면 8개 규칙(23)이 나옵니다. 답안에 간단한 2조건 4규칙 예시를 표로 그리면 확실한 점수를 받을 수 있어요.

소문제 2: 구조 기반 커버리지 비교

소문제 2로 넘어가면 구조 기반 테스트 커버리지 비교입니다. 화이트박스, 즉 코드 내부를 들여다보고 "이 코드가 얼마나 테스트되었는가"를 측정하는 거예요. 가장 중요한 건 포함 관계를 정확히 이해하는 거예요. 구문 커버리지가 가장 약하고, 분기 커버리지가 그 위, 그 다음이 조건/분기, 최상위가 MC/DC예요. "구분조MC"로 외우세요. 이 포함 관계도를 답안에 반드시 그려야 합니다.

구문 커버리지는 "모든 문장이 한 번씩 실행됐냐"를 보는 거고, 분기 커버리지는 "모든 if문의 True와 False를 다 탔냐"를 보는 거예요. True/False를 모두 타면 당연히 모든 문장도 실행되니까요.

조건 커버리지는 여기서 함정이 있어요. A AND B에서 {A=T, B=F}와 {A=F, B=T}를 테스트하면 조건 커버리지 100%인데, 분기 결과는 둘 다 False예요! True 경로를 안 탄 거죠! 이 예시를 반드시 답안에 쓰세요. "조건 커버리지 100%가 분기 커버리지 100%를 보장하지 않는다"는 걸 증명하는 핵심 예시거든요.

소문제 3: MC/DC

마지막 소문제 3, MC/DC입니다. Modified Condition/Decision Coverage의 약자로, 1992년 Boeing의 Chilenski와 Miller가 제안했어요. 3가지 요건이 있는데, "결조독"으로 외우세요. 결정 커버리지 충족, 조건 커버리지 충족, 그리고 독립적 영향 증명.

핵심 개념이 마스킹 효과의 방지인데요, AND 연산에서 하나가 False면 나머지가 뭐든 결과가 False잖아요? 그때 False인 조건이 다른 조건의 영향을 "가려버리는" 거예요. 쉽게 말하면, 여러분이 햇빛을 손으로 가리면 뒤에 달이 있는지 별이 있는지 안 보이잖아요? 그게 마스킹이에요. OR 연산에서는 반대로 하나가 True면 나머지를 마스킹해요.

MC/DC는 이런 마스킹이 발생하지 않는 상태에서만 각 조건의 영향을 테스트해요. A AND B에서 A의 독립적 영향을 보려면, B를 True로 고정해야 해요. 이 논리를 답안에 표로 정리하면 확실한 점수를 받습니다.

MC/DC의 큰 장점 하나 더. 효율성이에요. 조건이 N개일 때 최소 N+1개의 테스트 케이스만 필요합니다. 전체 조합은 2N개인데, MC/DC는 N+1이에요. 조건이 10개면 전체 조합은 1,024개지만 MC/DC는 11개만 있으면 되는 거죠.

적용 사례로 Boeing 787을 꼭 언급하세요. DO-178C Level A 인증을 위해 약 650만 라인 코드에 대해 MC/DC 100%를 달성했고, 테스트에만 2년, 비용 3,000억 원 이상 투입되었습니다.

결론 및 분량 배분

결론에서는 명세 기반과 구조 기반 테스트가 상호 보완적이라는 점, Safety-Critical System에서는 MC/DC 수준의 수학적 엄밀성이 필수라는 점을 강조하세요. 연관 토픽으로 V 모델을 언급하면 좋아요.

분량 배분은 서론 10%, 소문제 1이 25%, 소문제 2가 30%, 소문제 3이 25%, 결론 10%예요. 필수 도식은 커버리지 포함 관계도, MC/DC 독립적 영향 쌍 표, 명세 기반 3기법 비교표 이 세 가지입니다. 오늘도 수고 많으셨고, 다음 시간에 뵙겠습니다!

한 줄 정의

명세 기반 테스트는 요구사항으로부터, 구조 기반 테스트는 코드 구조로부터 테스트를 설계하며, MC/DC는 각 조건이 결정 결과에 독립적으로 영향을 미침을 증명하여 마스킹 효과를 방지하는 최상위 커버리지 기준이다.

핵심 키워드 맵

명세 기반 테스트 — 블랙박스
  • ├─ 동등 분할: 입력 도메인 → 파티션 분할 → 대표값 선택
  • ├─ 경계값 분석: 경계에서 65~70% 결함 집중, Off-by-One 오류
  • └─ 결정 테이블: 조건 조합 → 규칙 도출, 최대 2N 규칙
구조 기반 테스트 — 화이트박스
  • ├─ 구문 커버리지: 문장 실행 여부 (최저 수준)
  • ├─ 분기 커버리지: True/False 경로 (⊃ 구문)
  • ├─ 조건 커버리지: 개별 조건 T/F (분기 미보장 한계)
  • └─ MC/DC: 독립적 영향 증명 (최상위, N+1개 TC)
MC/DC
  • ├─ 정의: 각 조건이 결정에 독립적으로 영향
  • ├─ 마스킹 효과: AND → F가 마스킹, OR → T가 마스킹
  • ├─ 최소 TC: N+1개 (N=조건 수)
  • └─ 적용: DO-178C Level A, ISO 26262 ASIL D

두문자어 암기법

명세 기반 3대 기법: "동경결"

  • 동: 동등 분할(Equivalence Partitioning)
  • 경: 경계값 분석(Boundary Value Analysis)
  • 결: 결정 테이블(Decision Table)

  • 구: 구문(Statement) — 문장 실행 여부
  • 분: 분기(Branch) — True/False 경로
  • 조: 조건/분기 결합 — 개별 조건 + 분기
  • MC: MC/DC — 독립적 영향 증명 (최강)
